## Cara Menjalankan (urutan)

1. **Postgres** — jalan native sebagai Windows service (`postgresql-x64-18`), tidak perlu langkah apa pun kalau service Windows-nya sudah `Running`. Role `platform`/`platform` dan database `auth_service`, `company_service`, `rbac_service`, `audit_service`, `finance_service` sudah dibuat.
2. **Infra Kafka/Redis/MinIO/ClickHouse/Mosquitto** — coba dulu, JANGAN asumsikan Docker pasti rusak (lihat "Known Issues" #1, sesi 2026-07-12 ternyata sehat):
   ```
   cd infra
   docker compose up -d
   ```
   Kalau Docker Desktop error (`pipe error 500` / `request returned 500 Internal Server Error`), restart Docker Desktop dulu. Semua service Go **tetap bisa jalan tanpa ini** — publish/consume Kafka didesain best-effort (gagal → log warning, tidak crash), begitu juga MQTT untuk iot-service (lihat bagian "IoT Simulator" di bawah). Kalau image `bitnami/kafka:*` gagal pull ("not found"), itu karena Bitnami memindahkan image gratisnya ke `bitnamilegacy/*` (lihat komentar di `infra/docker-compose.yml`, sudah di-fix ke `bitnamilegacy/kafka:3.7.1`).
   **Penting**: kalau restart `audit-service` setelah Kafka baru naik (atau setelah topic BARU pertama kali dipakai — mis. sesi ini `iot.*`), restart SEKALI LAGI setelah topic-nya sudah pasti ada (lihat Known Issues #2, dikonfirmasi ulang sesi ini utk topic iot.*) — assignment pertama tiap reader per-topic bisa miss topic yang belum ada saat reader itu mulai.
   **ClickHouse butuh credential eksplisit** (sesi 2026-07-14/15) — image resmi ClickHouse MEMATIKAN akses network sama sekali untuk user `default` kalau `CLICKHOUSE_USER`/`CLICKHOUSE_PASSWORD` tidak diset di container (bukan cuma perlu password, request ditolak total). `infra/docker-compose.yml` sudah di-set `CLICKHOUSE_USER=default`/`CLICKHOUSE_PASSWORD=clickhouse` (dev-only, sama pola dengan MinIO `minioadmin`/`minioadmin`) — dw-service baca ini lewat env `CLICKHOUSE_USER`/`CLICKHOUSE_PASSWORD` (default sudah cocok, tidak perlu di-set manual). **Kalau container ClickHouse sudah pernah dibuat SEBELUM env var ini ditambahkan**, `docker compose up -d` biasa TIDAK cukup — perlu `docker compose up -d --force-recreate clickhouse` supaya env var baru benar-benar terpasang (container lama tetap jalan dengan config lama kalau cuma di-restart).

## ⚠️ Known Issues / Perlu Perhatian

1. ~~Docker Desktop error di mesin ini~~ — **update 2026-07-12: Docker ternyata sehat sesi ini**, `docker compose up -d` di `infra/` jalan bersih (Kafka, Redis, MinIO, ClickHouse, Kafka UI semua Up & reachable). Histori error `pipe error 500` di sesi-sesi sebelumnya kemungkinan besar karena beban memori dari project Docker lain di mesin ini (lihat memory `docker_instability_native_postgres`), bukan kerusakan permanen — coba dulu sebelum asumsi rusak. Dua bug nyata yang ditemukan & sudah di-fix di sesi ini:
   - `bitnami/kafka:3.7` sudah tidak bisa di-pull gratis (Bitnami memindahkannya ke langganan berbayar per 2025) → diganti `bitnamilegacy/kafka:3.7.1` di `infra/docker-compose.yml`.
   - `kafka-ui` sebelumnya mapped ke host port **8090**, sekarang bentrok dengan `production-service` milik project ini sendiri (ditambahkan sesi setelah docker-compose ini terakhir diedit) → dipindah ke port **8099**.
2. **Audit-service consumer group vs topic yang baru auto-created** — kalau `audit-service` di-restart lalu langsung dites dengan event dari topic yang BELUM PERNAH ADA sebelumnya di Kafka (baru auto-create saat consumer join), assignment generasi pertama bisa skip topic itu (event tersimpan di Kafka tapi tidak ke-assign ke consumer manapun, `kafka-consumer-groups.sh --describe` tidak menunjukkan baris untuk topic itu). Fix-nya cukup restart `audit-service` SEKALI LAGI setelah topic-nya sudah pasti ada — assignment generasi kedua akan meng-cover semua topic dengan benar. Ini murni race condition startup, bukan bug di kode `internal/consumer/consumer.go`.
   Terkait ini, sesi ini juga menambahkan topic modul Warehouse/Production/QC/Asset (`warehouse.*`, `production.*`, `qc.*`, `asset.*`) ke `Topics` list di `backend/services/audit-service/internal/consumer/consumer.go` — sebelumnya cuma topic dari Fase 1 + HR/Sales/Purchasing yang terdaftar, karena belum pernah bisa dites sampai Docker beneran jalan sesi ini.

   **Update sesi 2026-07-14 (IoT Simulator)**: dikonfirmasi ulang issue ini tetap reproduce untuk topic `iot.*` yang baru, dan mekanismenya sedikit beda dari deskripsi di atas — `Start()` di `consumer.go` sebenarnya spawn **satu Reader goroutine terpisah per topic** (bukan satu consumer group multi-topic), jadi bukan soal "assignment generasi pertama skip beberapa topic dari daftar", tapi tiap Reader individual yang mulai `ReadMessage()` SEBELUM topic-nya ada bisa gagal menemukan topic itu sampai di-restart. Fix-nya tetap sama (restart `audit-service` setelah topic pasti ada), tapi kalau ada waktu untuk benerin permanen, lihat rekomendasi #4 di "Next Steps".

### Production module — detail implementasi (untuk konteks kalau ada bug/lanjutan)

- `bill_of_materials`/`bom_lines` dan `work_orders`/`work_order_lines` hidup di `production-service` (baru, port 8090, db `production_service`). `product_id` (produk jadi) dan `component_product_id` (bahan baku) menunjuk ke tabel `products` milik warehouse-service TANPA FK fisik (beda database) — production-service percaya `product_id` yang dikirim frontend (yang ambil daftar produk langsung dari `GET /api/warehouse/products`, sama seperti pola dropdown produk di halaman Warehouse).
- `work_order_lines` men-snapshot `bom_lines.quantity_per_unit * quantity_planned` saat work order dibuat (bukan re-lookup BOM saat completed), supaya perubahan BOM belakangan tidak mengubah kebutuhan komponen WO yang sudah berjalan.
- Alur status: `DRAFT` → `POST .../start` → `IN_PROGRESS` → `POST .../complete` (body `{quantity_produced}`) → `COMPLETED`. ### Forecasting & Anomaly Detection (AI & BI, pelengkap) — detail implementasi

- Dua endpoint baru ditambahkan ke `ai-bi-service` yang sudah ada (bukan service baru): `GET /forecasting/summary` dan `GET /anomaly-detection/scan`, di file `internal/httpapi/forecasting.go` dan `internal/httpapi/anomaly.go`. Masih 100% stdlib, masih stateless, masih tanpa database.
- **Forecasting** = regresi linear sederhana (least squares manual, bukan library) atas agregasi bulanan: (1) total revenue sales per bulan dari `GET /sales-orders`, (2) total kuantitas stok (semua produk/gudang digabung) direkonstruksi dari running cumulative sum `stock_movements` (IN +, OUT -) diurutkan kronologis. Default 6 bulan histori + proyeksi 3 bulan ke depan (bisa diubah lewat query param `history_months`/`forecast_months`). Bulan tanpa transaksi di-carry-forward dari bulan sebelumnya (`fillMonthlySeries`) supaya seri level (stok) masuk akal; untuk seri revenue ini berarti bulan tanpa order tetap 0, bukan meniru bulan sebelumnya (revenue bukan level, wajar 0 kalau tidak ada order).
  **Batasan yang perlu diingat**: `GET /stock-movements` di warehouse-service dibatasi 200 baris terbaru (lihat kode warehouse-service), jadi rekonstruksi kuantitas stok cuma akurat untuk data dengan histori mutasi < 200 baris. Untuk skala production sungguhan, endpoint itu perlu pagination dulu sebelum forecasting stok bisa diandalkan penuh.
- **Anomaly Detection** = z-score per domain (bukan model ML): kumpulkan semua nilai (total_amount sales order / total_amount purchase order / quantity stock movement) per company, hitung mean+stddev populasi, tandai yang `|z| >= threshold_z` (default 2.0, bisa diubah lewat query param). Domain dengan kurang dari 3 data point atau stddev=0 (semua nilai sama) dilewati diam-diam (tidak cukup data untuk anomali yang bermakna). Response selalu sertakan `mean`/`stddev`/`z_score` mentah supaya user bisa menilai sendiri kewajaran flag-nya, bukan cuma label "anomali" tanpa konteks.
- Frontend: `ForecastingPage.jsx` pakai komponen custom `ForecastLineChart.jsx` (SVG murni, tanpa library chart baru) — garis solid untuk histori, garis putus-putus untuk proyeksi (HUE SAMA, dibedakan lewat gaya garis bukan warna, sesuai prinsip "identitas lewat lebih dari sekadar warna"), lengkap dengan hover crosshair+tooltip (posisi tooltip di-clamp supaya tidak meluber di titik data pertama/terakhir). `AnomalyDetectionPage.jsx` pakai `DataTable` yang sudah ada (bukan komponen baru) karena ini pada dasarnya daftar/tabel, bukan chart.
- Sudah diverifikasi end-to-end: dibuat 4 sales order sintetis (100, 110, 95, 5.000.000) khusus untuk MEMBUKTIKAN logika z-score benar bekerja (lihat "Data Demo" di bawah) — z-score untuk order 5 juta itu terhitung ~1.9999 (di bawah threshold default 2.0 tapi kena kalau threshold diturunkan ke 1.5, dikonfirmasi lewat curl manual sebelum test browser). Playwright kemudian konfirmasi kedua halaman render tanpa error, chart forecasting menampilkan histori+proyeksi dengan benar, dan halaman anomaly menampilkan empty-state yang benar untuk threshold default.

**Yang dikerjakan:**

2. **`frontend/web/Dockerfile`** baru — multi-stage `node:20-alpine` (build) → `nginx:1.27-alpine` (serve static). `ARG VITE_API_BASE_URL` (default `http://localhost:8079`) di-`ENV`-kan sebelum `npm run build` karena Vite bake env `VITE_*` ke bundle JS saat build time, bukan runtime — beda dari service Go yang baca env var saat container start. `nginx.conf` custom (`try_files $uri $uri/ /index.html`) untuk SPA fallback (React Router). `.dockerignore` (`node_modules`, `dist`, `.git`) supaya build context tidak ikut node_modules yang besar.
3. **2 bug nyata ditemukan & diperbaiki**, keduanya baru ketahuan setelah benar-benar mencoba `docker compose build` (bukan cuma nulis YAML/Dockerfile tanpa diuji):
   - **Go version mismatch**: 4 Dockerfile Fase 1 (auth/company/rbac/audit) pakai `golang:1.22-alpine` padahal `go.mod`-nya deklarasi `go 1.25.0` — dibiarkan sebelumnya karena `GOTOOLCHAIN=auto` diam-diam auto-download toolchain baru saat build (jalan tapi boros & rapuh di lingkungan offline/CI). Dibenarkan ke `golang:1.25-alpine`. (`api-gateway` sendiri declare `go 1.22` jadi Dockerfile-nya sudah benar, tidak disentuh versi Go-nya — cuma `EXPOSE 8080` yang salah dibenarkan ke `8079` sesuai `PORT` default sebenarnya, dan `COPY go.mod ./` dilengkapi jadi `COPY go.mod go.sum ./` untuk konsistensi.)
   - **8 dari 9 module service TIDAK PUNYA `go.sum` sama sekali** (cuma `finance-service` yang punya, dari awal project). Ini lolos tanpa masalah selama ini karena `go build`/`go test` selalu dijalankan lewat `backend/go.work` (workspace mode), yang bisa resolve dependency dari module cache tanpa perlu `go.sum` per-modul individual. Efek samping menarik: `ai-bi-service` ternyata BUKAN pure-stdlib seperti komentar lama di `config.go` mengklaim (`"ai-bi-service tidak punya database sendiri..."`) — test file-nya pakai `github.com/google/uuid` untuk generate ID company dummy, jadi `go.mod`-nya sekarang punya 1 `require` (uuid v1.6.0). Ini murni penambahan dependency test yang sudah lama dipakai tapi belum pernah dideklarasikan eksplisit, bukan perubahan perilaku.
4. **`infra/docker-compose.yml` diperluas** — semua 14 service Go + frontend ditambahkan sebagai `services:` yang bisa di-`build`, dengan wiring jaringan container yang benar (bukan cuma copy-paste default `localhost` dari `config.go`):
   - **Postgres** tetap native (bukan container, sesuai keputusan lama) — tiap service app terhubung lewat hostname `host.docker.internal` (bukan `localhost`, yang di dalam container merujuk ke dirinya sendiri), plus `extra_hosts: ["host.docker.internal:host-gateway"]` di tiap service supaya juga jalan di Docker Desktop non-Windows/Mac.
   - **Kafka dapat listener kedua** (`INTERNAL://:29092`, advertised sebagai `kafka:29092`) khusus untuk client di dalam jaringan compose, terpisah dari listener lama (`PLAINTEXT://:9092`, advertised `localhost:9092`) yang tetap dipakai proses native (`go run`). Tanpa ini, container client yang bootstrap ke `kafka:9092` akan dapat metadata response yang bilang "reconnect ke localhost:9092" — yang di dalam container itu berarti dirinya sendiri, bukan broker Kafka — dan produce/consume gagal diam-diam sesudah koneksi awal. `kafka-ui` ikut dipindah ke listener internal (`kafka:29092`).
   - URL lintas-service (`FINANCE_SERVICE_URL`, `WAREHOUSE_SERVICE_URL`, dst di `api-gateway`/`hr-service`/`sales-service`/`purchasing-service`/`production-service`/`ai-bi-service`) pakai nama container (mis. `http://finance-service:8085`), bukan `localhost`.
   - `frontend`nya sendiri BUKAN pakai container-network URL untuk API — `VITE_API_BASE_URL` tetap `http://localhost:8079` (default) karena yang memanggil API adalah browser user (client-side JS), bukan container lain di jaringan compose.

1. **ClickHouse butuh auth eksplisit** — image resmi `clickhouse/clickhouse-server` MEMATIKAN akses network total untuk user `default` kalau `CLICKHOUSE_USER`/`CLICKHOUSE_PASSWORD` tidak diset di container (log entrypoint: `"neither CLICKHOUSE_USER nor CLICKHOUSE_PASSWORD is set, disabling network access for user 'default'"`) — beda dari semua infra lain di project ini (Kafka/Redis/Mosquitto benar-benar no-auth). Fix: `infra/docker-compose.yml` di-set `CLICKHOUSE_USER=default`/`CLICKHOUSE_PASSWORD=clickhouse` (dev-only, pola sama `minioadmin`/`minioadmin`), `dw-service` config dapat `CLICKHOUSE_USER`/`CLICKHOUSE_PASSWORD` env var (default cocok). Container yang SUDAH ADA sebelum env var ini ditambahkan perlu `docker compose up -d --force-recreate clickhouse`, bukan cuma `up -d` biasa.
2. **`float64` tidak bisa di-bind ke kolom `Decimal` ClickHouse** — ditemukan lewat test asli (`converting float64 to Decimal(18, 2) is unsupported`), bukan dari baca dokumentasi dulu. Driver `clickhouse-go/v2` cuma dukung `decimal.Decimal` (`github.com/shopspring/decimal`, sudah transitive dependency, dijadikan direct) untuk kolom `Decimal(P,S)`, baik arah insert (`Append`) maupun read (`Scan`) — `float64` cuma valid untuk kolom `Float64`. Fix: helper `toDecimal(f float64) decimal.Decimal` di titik `Append` (row struct tetap `float64`, konversi cuma di boundary ClickHouse), dan test diupdate scan ke `decimal.Decimal` juga.
3. **Race DDL Postgres antar test package** — `internal/httpapi` dan `internal/etl` awalnya sama-sama migrate ke `dw_service_test`, `go test ./...` menjalankan keduanya sebagai proses paralel terpisah, `CREATE TABLE`/`CREATE EXTENSION IF NOT EXISTS` race (`duplicate key value violates unique constraint pg_type_typname_nsp_index`) — pola persis sama dengan bug serupa di sesi IoT (httpapi vs ingest), sekarang dikonfirmasi ini pola UMUM bukan kebetulan sekali. Fix: `internal/etl` pakai database source test terpisah dari modul lain manapun (`dw_service_test`, sengaja TIDAK bentrok karena cuma satu package yang butuh source Postgres — cukup satu `TestMain` di `internal/etl`, `internal/httpapi` tidak punya test sendiri sesi ini).

**Test**: `internal/etl` — `TestMain` bikin schema MINIMAL sendiri (bukan import migration finance-service/sales-service/warehouse-service asli — sengaja dihindari supaya tidak ada dependency test-time lintas modul) yang meniru PERSIS kolom yang dipakai extract SQL, plus konek ClickHouse (SKIP kalau tidak reachable, pola sama seperti SKIP-if-no-Postgres). 5 test: 1 extract+load per fact (finance/sales/inventory) + 1 test khusus DRAFT journal entry (buktikan status apa pun tersinkron, bukan cuma POSTED) + 1 test idempotency (`SyncFinance` dipanggil 2x, hitung baris via `FINAL` harus tetap 1 per `line_id`).
